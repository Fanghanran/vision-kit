# 规则引擎（Rule Engine） — 设计书

## 1. 模块职责

规则引擎是 SentinelMind 处理层的核心组件，负责将检测与追踪结果转化为有意义的事件告警。其职责包括：

1. **规则加载**：从 YAML 配置文件和内置规则注册表加载规则定义，解析 conditions 和 actions 字段，生成可执行的规则实例。
2. **规则评估**：对每一帧的追踪结果运行所有适用规则，判断是否触发事件。
3. **三层防线过滤**：在规则判定通过后，依次执行滑动窗口去重、冷却时间、时间窗口三重过滤，防止误报泛滥。
4. **事件生成**：通过防线的规则触发结果被封装为 Event 对象，传递给下游的告警处理管线。
5. **规则热重载**：监听规则配置文件变化，自动重新加载规则，无需重启进程。
6. **内置规则注册**：维护内置规则的注册表，统一内置规则和自定义规则的加载入口。

规则引擎遵循"声明式优先、函数式扩展"的设计理念：80% 的常见场景通过 YAML 声明式配置完成，20% 的复杂场景通过 Python 类扩展实现。

## 2. 对外接口

### 2.1 RuleProtocol（规则协议接口）

所有规则（内置和自定义）必须实现此 Protocol。

| 成员 | 类型 | 签名 | 说明 |
|------|------|------|------|
| name | 属性 | str | 规则名称，全局唯一，用于日志和告警关联 |
| camera_ids | 属性 | list[str] 或 None | 适用摄像头 ID 列表，None 表示适用于所有摄像头 |
| evaluate | 方法 | (tracks: list[Track], frame: numpy.ndarray, context: dict) -> Event 或 None | 评估当前帧是否满足触发条件，返回 None 表示未触发 |
| reset | 方法 | () -> None | 重置规则内部状态（滑动窗口计数器、缓存等），用于热重载或手动重置 |

**evaluate 参数说明**：

- `tracks`：当前帧的所有追踪目标列表，每个元素为 Track 对象，包含 track_id、class_name、bbox、trajectory、velocity 等字段。
- `frame`：当前帧的原始图像（numpy BGR 数组），用于截图或视觉判断。
- `context`：上下文字典，包含 camera_id、camera_name、timestamp、frame_id 等元信息。

**evaluate 返回值**：满足条件时返回 Event 对象（包含 event_type、camera_id、severity、metadata 等），未满足条件返回 None。

### 2.2 RuleEngine 类

| 方法 | 签名 | 返回值 | 说明 |
|------|------|--------|------|
| __init__ | (config: dict) -> None | — | 初始化引擎，传入 rules 配置段 |
| load_rules | (rules_dir: str) -> None | None | 加载指定目录下所有 YAML 规则文件 |
| evaluate | (tracks: list[Track], frame: numpy.ndarray, context: dict) -> list[Event] | 事件列表 | 对当前帧运行所有适用规则，返回触发的事件列表 |
| get_rule | (name: str) -> RuleProtocol 或 None | 规则实例 | 按名称获取已加载的规则 |
| unload_rule | (name: str) -> bool | 是否成功 | 卸载指定规则，重置其内部状态并从引擎中移除 |
| reload | () -> None | None | 重新加载所有规则文件，替换现有规则实例 |
| register_builtin | (rule_class: type) -> None | None | 注册一个内置规则类 |
| list_rules | () -> list[dict] | 规则摘要列表 | 返回所有已加载规则的摘要信息（name、camera_ids、来源类型） |

### 2.3 三层防线相关

| 类/方法 | 签名 | 说明 |
|---------|------|------|
| DefenseFilter.__init__ | (cache: CacheProtocol, config: dict) -> None | 初始化防线，注入缓存实例和配置 |
| DefenseFilter.check | (rule_name: str, camera_id: str, event: Event) -> bool | 执行三层过滤，返回 True 表示允许通过（应生成告警） |
| DefenseFilter._sliding_window_check | (key: str, window_size: int) -> bool | 第一层：滑动窗口去重检查 |
| DefenseFilter._cooldown_check | (key: str, cooldown_seconds: int) -> bool | 第二层：冷却时间检查 |
| DefenseFilter._time_window_check | (time_windows: list[dict]) -> bool | 第三层：时间窗口检查 |

## 3. 内部逻辑

### 3.1 规则加载流程

1. 扫描 `rules_dir` 目录下所有 `.yaml` 和 `.yml` 文件。
2. 对每个文件执行 YAML 解析，校验必填字段（name、conditions）。
3. 根据配置判断规则类型：
   - 若存在 `module` 和 `class` 字段，则为 Python 扩展规则。通过 `importlib` 动态导入模块，实例化指定类。
   - 若仅有 `conditions` 字段，则为 YAML 声明式规则。根据 conditions 中的 `type` 字段选择对应的内置评估器（object_in_zone、count_line、zone_empty 等），自动生成规则实例。
4. 对每个规则实例执行一次 `reset()` 初始化。
5. 注册到引擎内部的规则字典（key 为规则名称）。
6. 加载过程中任何语法错误或校验失败，记录错误日志并跳过该文件，不影响其他规则的加载。

### 3.2 规则评估流程

1. ActionThread 从 ResultQueue 取出一帧的追踪结果。
2. 调用 `RuleEngine.evaluate(tracks, frame, context)`。
3. 引擎遍历所有已加载规则：
   - 检查该规则的 `camera_ids` 属性，若不为 None 且当前 camera_id 不在列表中，跳过。
   - 调用规则的 `evaluate()` 方法。
   - 若返回 Event 对象，送入三层防线过滤。
4. 三层防线逐层检查：
   - **滑动窗口**：使用缓存记录该规则+摄像头的连续触发帧数。若连续触发帧数 < 窗口大小（默认 5 帧），记录但不放行。达到阈值后放行，重置计数器。
   - **冷却时间**：使用缓存记录该规则+摄像头的上次放行时间。若距上次放行时间 < 冷却时间（默认 300 秒），拒绝放行。
   - **时间窗口**：检查当前时间是否在规则配置的时间窗口内。若规则未配置时间窗口则放行。支持配置多个时间段（如 22:00-06:00）和星期几生效。
5. 三层全部通过的事件加入结果列表。
6. 返回所有触发的事件列表（可能为空）。

### 3.3 三层防线详细逻辑

#### 滑动窗口去重（第一层）

- 缓存 key 格式：`rule:{rule_name}:camera:{camera_id}:sliding_window`
- 缓存 value：连续触发帧数（整数）
- 逻辑：每次触发时将计数器加 1。若计数器 >= 配置的窗口大小（默认 5），判定为有效触发，放行并重置计数器为 0。若某次未触发，重置计数器为 0。
- 防误报原理：单帧检测偶然出现的误报无法连续 5 帧出现，只有稳定的异常行为才能通过此层。

#### 冷却时间（第二层）

- 缓存 key 格式：`rule:{rule_name}:camera:{camera_id}:cooldown`
- 缓存 value：上次放行的时间戳（Unix 秒）
- 逻辑：读取缓存中的上次放行时间。若不存在或距当前时间 >= 冷却时间（默认 300 秒），放行并更新缓存。否则拒绝。
- 防重复原理：同一摄像头同一规则在冷却期内只允许一次告警，避免同一事件反复通知。

#### 时间窗口（第三层）

- 数据来源：规则 YAML 中的 `time_windows` 配置段。
- 逻辑：取当前本地时间，检查是否在任一配置的时间窗口内。
  - 窗口格式：`start: "HH:MM"`, `end: "HH:MM"`, `days: [0,1,2,3,4,5,6]`（0=周一，6=周日）
  - 支持跨午夜窗口（如 22:00-06:00）：若 end <= start，表示跨午夜。
  - 未配置 time_windows 的规则，此层默认放行。
- 防无关时段误报：某些规则只在特定时段有意义（如离岗检测只在工作时间）。

### 3.4 规则热重载流程

1. 启动一个文件监控线程（或使用定时扫描，间隔 5 秒）。
2. 记录所有已加载规则文件的最后修改时间（mtime）。
3. 定期检查每个文件的 mtime 是否变化。
4. 若检测到变化：
   - 重新解析该 YAML 文件。
   - 校验新规则配置。
   - 若校验通过：调用旧规则的 `reset()` 清理状态，替换为新规则实例。
   - 若校验失败：保留旧规则不变，记录错误日志。
5. 若检测到新增文件：加载新规则。
6. 若检测到文件删除：卸载对应规则。
7. 整个重载过程对正在处理的帧无影响，新规则在下一帧评估时生效。

### 3.5 内置规则注册

1. 引擎初始化时扫描 `rules/builtin/` 目录下所有 Python 文件。
2. 找到所有继承自基础规则类或实现 RuleProtocol 的类。
3. 调用 `register_builtin(rule_class)` 注册到内置规则注册表。
4. YAML 声明式规则的 conditions.type 映射到对应的内置评估器。
5. 注册表是一个字典，key 为 type 名称（如 intrusion、absence、crowd），value 为评估器类。

## 4. 依赖关系

| 依赖模块 | 依赖方向 | 说明 |
|----------|----------|------|
| config | 规则引擎 → 配置管理 | 读取 rules_dir 路径、热重载开关等配置 |
| cache | 规则引擎 → 缓存层 | 三层防线的状态存储依赖缓存（滑动窗口计数、冷却时间戳） |
| core/types | 规则引擎 → 数据类型 | 使用 Track、Event 等数据模型 |
| core/tracker | 规则引擎 ← 追踪器 | 接收 Track 列表作为输入 |
| actions/notifier | 规则引擎 → 通知器 | 通过 Event 间接触发通知，规则引擎不直接调用 |
| storage/database | 规则引擎 → 数据库 | 无直接依赖，Event 由下游写入数据库 |

**依赖注入方式**：RuleEngine 通过构造函数接收 config 字典和 CacheProtocol 实例，不直接依赖具体缓存实现。内置规则通过注册表机制解耦，新增规则无需修改引擎代码。

## 5. 配置项

### 5.1 全局规则引擎配置（settings.yaml 中的 rules 段）

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| rules_dir | str | configs/rules | 规则文件目录路径 |

| hot_reload_interval | int | 5 | 热重载扫描间隔（秒） |
| default_window_size | int | 5 | 滑动窗口默认大小（帧数） |
| default_cooldown | int | 300 | 冷却时间默认值（秒） |

### 5.2 单条规则 YAML 配置格式

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|

| description | str | 否 | 规则描述 |
| camera_ids | list[str] | 否 | 适用摄像头列表，缺省为全部摄像头 |

| conditions | list[dict] | 是 | 触发条件列表，支持多条件组合 |

| time_windows | list[dict] | 否 | 生效时间窗口列表 |
| module | str | 否 | Python 扩展规则的模块路径 |
| class | str | 否 | Python 扩展规则的类名 |
| params | dict | 否 | 传递给扩展规则的额外参数 |

### 5.3 time_windows 子配置格式

| 字段 | 类型 | 说明 |
|------|------|------|
| start | str | 开始时间，格式 HH:MM（24 小时制） |
| end | str | 结束时间，格式 HH:MM（支持跨午夜） |
| days | list[int] | 生效的星期几，0=周一，6=周日，缺省为全部 |

## 6. 错误处理

### 6.1 规则加载错误

| 错误场景 | 处理方式 | 是否阻断启动 |
|----------|----------|-------------|
| YAML 语法错误 | 记录 ERROR 日志，跳过该文件，继续加载其他规则 | 否 |
| 缺少必填字段（name 或 conditions） | 记录 ERROR 日志，跳过该文件 | 否 |
| Python 扩展模块导入失败 | 记录 ERROR 日志，跳过该规则 | 否 |
| 扩展类未实现 RuleProtocol | 记录 ERROR 日志，跳过该规则 | 否 |
| 规则名称冲突（重复） | 记录 WARNING 日志，后加载的覆盖先加载的 | 否 |
| 规则目录不存在 | 记录 WARNING 日志，使用内置规则运行 | 否 |

### 6.2 规则评估错误

| 错误场景 | 处理方式 | 是否影响其他规则 |
|----------|----------|-----------------|
| 单条规则 evaluate 抛出异常 | 捕获异常，记录 ERROR 日志，跳过该规则，继续评估其他规则 | 否 |
| Track 数据格式异常 | 在引擎层做防御性校验，异常 Track 被过滤 | 否 |
| 三层防线缓存访问失败 | 降级为放行（宁可多报不漏报），记录 WARNING | 否 |
| 时间窗口配置解析错误 | 该时间窗口条件视为不满足，记录 WARNING | 否 |

### 6.3 热重载错误

| 错误场景 | 处理方式 |
|----------|----------|
| 新规则文件校验失败 | 保留旧规则不变，记录 ERROR |
| 文件监控线程异常退出 | 记录 CRITICAL 日志，尝试重启监控线程 |
| 规则替换过程中发生异常 | 保留旧规则不变，记录 ERROR，不影响正在处理的帧 |

## 7. 设计决策

### 7.1 为什么选择声明式 + 函数式混合设计

纯声明式（如简单 JSON 规则）表达能力有限，无法处理复杂的时序逻辑（如"物体停留 5 分钟"需要内部状态）。纯函数式（全部用 Python 代码）对非开发人员不友好，配置维护成本高。

声明式 YAML 覆盖 80% 的常见规则（闯入检测、人数统计等），用户只需编辑 YAML 即可配置新规则。Python 扩展覆盖 20% 的复杂场景（如叉车超速、特殊行为识别），开发者编写一个类即可接入。两者共享同一套 RuleProtocol 接口和三层防线机制。

### 7.2 为什么三层防线是共享的而非规则自定义的

如果每个规则自己实现去重和冷却逻辑，会导致：
- 实现不一致（有的规则忘了做冷却，导致重复告警泛滥）。
- 参数风格不统一（有的用帧数，有的用秒数）。
- 无法统一优化（缓存策略、key 管理无法集中处理）。

共享的 DefenseFilter 确保所有规则的行为一致，且可以在引擎层统一优化缓存策略。规则只需要关注"什么条件算触发"，不需要关心"触发后如何过滤"。

### 7.3 为什么滑动窗口用帧数而不用时间

帧数与推理频率直接相关，比时间更稳定。如果用时间（如"持续 2 秒"），当帧率波动时（摄像头抖动、GPU 负载变化），同样的时间窗口内可能有不同数量的帧参与判定，导致行为不一致。使用帧数（如"连续 5 帧"）则无论帧率如何波动，都需要固定数量的独立观测确认，判定结果更稳定。

### 7.4 为什么热重载采用文件扫描而非 watchdog

watchdog（inotify/kqueue）依赖操作系统机制，在不同平台上行为不一致（尤其在 Docker/NFS 挂载的目录下不可靠）。简单的文件扫描（每 5 秒检查 mtime）实现简单、跨平台可靠、对性能无影响（规则文件数量少，扫描开销极小）。

### 7.5 为什么规则评估是同步的

规则评估是纯计算操作（遍历 Track 列表、几何计算、缓存读写），执行时间在微秒到毫秒级别，无需异步化。将规则评估放在 ActionThread 中同步执行，保证了帧的处理顺序和状态一致性（滑动窗口的连续性依赖顺序处理）。LLM 调用和通知发送才是耗时操作，通过异步任务处理。

### 7.6 Event 与 Alert 的分离

规则引擎产出 Event（"发生了什么"），下游管线决定是否生成 Alert（"需要通知谁"）。一个 Event 可能因为三层防线被过滤掉而不产生 Alert；同一个 Event 也可能产生多个 Alert（通知到不同渠道）。这种分离让规则引擎保持纯粹的"判定"职责，不掺杂通知逻辑。
